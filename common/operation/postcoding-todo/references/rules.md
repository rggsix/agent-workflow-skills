# 规则

一句话定位：定义编码后 ToDo 校正的读取与写入边界。

## 必须

1. 通过 `todo-closure` 完成闭环。
2. 仅基于本次执行结果进行新增todo。
3. 复盘实际改动，补充遗漏/隐患/优化点的 ToDo，按风险分级写入：
   - P0: 不闭环会导致线上风险/阻塞发布
   - P1: 短期必须修，但不立即阻塞
   - P2: 可选优化
4. 若归档项含 `@path:`，归档记录中需保留原路径信息。
5. 写入必须通过 `todo` skill。
6. 若无新增或完成项，给出检查路径(文件级别即可)，以及不标记ToDo的理由。

## todo内容参考

### P0

- 定义了事件但没有处理者
- 占位方法
- 占位属性(返回空值/空串/占位对象等)
- mock方法，需要上线前移除的测试代码
- 新增开关或配置，但默认值会导致线上行为错误
- 关键链路仅完成半套改动，剩余部分未接通会直接报错或数据错乱
- 临时绕过校验/权限/崩溃保护，明确不能带到发布

### P1

- 已接入主流程，但缺少必要兜底，如空态/失败态/超时处理
- 新增字段或接口适配完成，但调用方、埋点、单测、迁移脚本未补齐
- 为了赶本次目标先写死参数、文案、分支，后续必须回收
- 已留下明确技术债，短期不处理会持续放大改动成本
- 存在 TODO/FIXME/占位分支，但当前不至于阻塞发布
- 代码路径已经能走通，但和原有约定不一致，后续容易引发误用

### P2

- 性能优化，如重复请求、重复计算、可缓存但未缓存
- 结构优化，如抽公共组件、收敛重复逻辑、拆分过长方法
- 可读性优化，如命名不清、注释缺失、局部实现可更清晰
- 辅助质量建设，如补充非关键测试、补文档、补开发期脚本

## 不新增 ToDo 的情况

- 纯主观“代码还不够优雅”，但没有明确后续动作
- 与本次改动无关的泛化优化建议
- 已在 `active.md` 中存在语义相同任务；这类应更新已有项，不重复新增

## 禁止

- 跳过 `todo-closure` skill 调用
- 不做新规划。
- 不修改 plan.md。
- 不直接写 todo 文件。
- 只做检查与 ToDo 更新，不修改代码。
